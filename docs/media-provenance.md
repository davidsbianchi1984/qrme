# Media provenance & watermarking

**Status: deliberately out of scope for v1 — documented here so the decision is
explicit, with a concrete design for when it lands.**

QRME v1 profiles produce **text**. The platform does not generate images, audio,
or video, so there is no synthetic media file to watermark in v1, and shipping a
watermarking pipeline now would be dead code guarding a capability that doesn't
exist yet. The v1 transparency guarantees that *do* apply to text are already
implemented: every profile is labelled AI, replies pass moderation, and
`GET /profiles/{id}/transparency` discloses the synthetic nature of the profile.

## When generated media arrives

Any future feature that has a profile emit an image / audio / video clip will
attach provenance at creation, following the open
[C2PA](https://c2pa.org/) content-credentials standard rather than a bespoke
scheme:

- **Signed manifest** — each generated asset carries a C2PA manifest asserting
  *"AI-generated by QRME"*, the profile id, and the creation timestamp, signed
  with the deployment's signing key.
- **Durable watermark** — a perceptual/invisible watermark (e.g. SynthID-class)
  embedded in the pixels/samples so provenance survives re-encoding and
  screenshotting, not only metadata stripping.
- **Verify endpoint** — `GET /media/{id}/provenance` returns the manifest and
  verification status, so a downstream viewer can confirm origin.
- **Vaulted originals** — the signed original is sealed in PDI, so the
  authoritative provenance record is tamper-evident and auditable.

None of this blocks the v1 release; it is gated on the media-generation feature
itself, which is a separate, later effort.
